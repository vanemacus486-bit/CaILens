# CaILens

> 一个克制的周历，正在长成一本时间日志。

CaILens 是一个本地优先的时间记录工具，灵感来自《奇特的一生》里柳比歇夫坚持了 56 年的"时间统计法"。从一个精良的周视图开始，逐步向上构建分类系统、周统计、ICS 导入——没有账号、没有服务器、没有数据上报。

![CaILens 周视图](https://github.com/user-attachments/assets/f8e2a65e-a9ce-44a9-929f-ab4a790c4b84)

> **状态：** 持续开发中。v2 已发布，包含分类系统和周统计。ICS 导入 + 关键词自动分类 + 独立统计页已完成。**设置页和统计页功能可用了，但 UI 还不够精致，后续还会优化。**

## 为什么做这个

市面上几乎所有日历工具都在解决同一个问题：把未来安排进格子里。提醒你开会、收会议邀请、设闹钟。但这和"理解自己的时间"是两件事。

柳比歇夫从 26 岁开始记录自己每一天每一小时的去向，整整 56 年，直到去世。他不是为了"提高效率"，而是为了真的看清自己一辈子的时间用在哪里了。

CaILens 想做后者。从最基础的部分开始——日历网格本身必须足够好用，后面的统计、回顾、分类才有意义。

原则：

- **本地优先**。数据存在浏览器的 IndexedDB 里，没有账号、没有服务器、没有数据上报。
- **直接操作**。点、拖、拉、打字。弹窗是最后的手段。
- **安静的设计**。克制的配色，不催促、不评价、不鼓励。用完即走。

## 功能

### 日历模块（v1，稳定）

- **24 小时一屏可见**——周一到周日，每天 0-24 点完整显示。
- **点击空白格创建事件**——选好时间段后直接弹编辑卡片。
- **拖拽事件移动**——可以跨天拖拽。
- **拖拽上下边缘改变时长**——不用开任何弹窗。
- **拖动时实时预览**——事件跟着鼠标走，目标时间实时更新，60fps。
- **双击事件 inline 改名**——不开弹窗，点完即改。
- **右键菜单**——删除、复制、改颜色。
- **当前时间红线**——仅今日列显示，每分钟更新。
- **重叠事件并排**——自动水平排列，不遮挡。
- **浅色 / 深色模式**。

### 分类与统计（v2，稳定）

- **6 个固定分类**（核心工作、辅助工作、必要事务、阅读学习、休息、其他），每个有独立颜色，用户可分别修改中英文名称。
- **分类选择器**——创建/编辑事件时必须选择一个分类。
- **周统计**——独立统计页（侧边栏 BarChart3 图标入口），展示本周总记录小时数和各分类占比条。重叠时间先去重再算百分比。
- **双语 UI**——中文和英文，在设置中切换。
- **侧边栏导航**——替代了 v1 的顶部导航。今天、上一周/下一周、统计、设置、ICS 导入都在这里。

### 导入（v2，稳定）

- **.ics 文件导入**——通过 ical.js 解析 RFC 5545 日历文件。全天事件和重复事件会自动跳过，预览中显示跳过数量。
- **关键词自动分类**——每个分类可在设置中配置关键词。导入时按事件标题匹配关键词（大小写不敏感、子串匹配），命中则自动归类；未命中则 fallback 到你选择的默认分类。第一匹配优先（分类顺序 accent → stone）。
- **关键词修改自动重新归类**——修改任意分类的关键词后，全库所有已有事件会自动重跑匹配，更新颜色和分类。

### 数据

- **本地持久化**——IndexedDB（通过 Dexie）。刷新页面、关闭浏览器，数据都在。
- **Schema 迁移**——版本化升级（v1→v3）自动执行。
- **6 种事件颜色**（accent/sage/sand/sky/rose/stone）——暖色调，扫一眼就能看出时间分布。

## 接下来要做什么

大致优先级：

- **增强统计**——柳比歇夫式的分类汇总，支持任意时间段（月度、自定义范围）。
- **日志 / 备注**——在时间线上附加每日或每周的回顾。
- **导出**——JSON 和 ICS 导出，数据随时可以带走。
- **全局搜索**所有事件。
- **可选同步**——依然本地优先，同步是 opt-in，不是默认。
- **UI 打磨**——设置页和统计页目前功能优先，视觉还需要再调整。

## 怎么跑起来

需要 **Node 20 以上**和 **npm**。

```bash
git clone https://github.com/vanemacus486-bit/CaILens.git
cd CaILens
npm install
npm run dev
```

然后打开 Vite 启动后打印的地址（通常是 `http://localhost:5173`）。

### 其他命令

```bash
npm run dev          # 启动开发服务器
npm run build        # 类型检查 + 生产构建
npm run preview      # 本地预览构建结果
npm run test         # 跑一次单元测试（530+ 测试，25 个测试文件）
npm run test:watch   # 监听模式跑测试
npm run lint         # 跑 ESLint
```

## 架构

代码按层划分，依赖方向严格单向：

```
domain/  ──→  data/  ──→  stores/  ──→  features/ + components/
 (纯逻辑)    (Repository)   (Zustand)        (UI)
```

- **`domain/`** —— 纯类型和业务规则。不依赖 React、不依赖 IndexedDB、没有副作用。容易单元测试。
- **`data/`** —— Dexie 的 schema 和 repository。整个项目里唯一直接和 IndexedDB 打交道的地方。
- **`stores/`** —— Zustand store，包装数据层，对 UI 暴露 selectors 和 actions。
- **`features/` + `components/`** —— React 组件。它们从不直接碰数据层。

几个值得一提的实现细节：

- **拖拽系统**直接基于 Pointer Events 写，没用现成库。命中检测和吸附是对着布局网格算的，不是对着 DOM。
- **渲染性能**——大量用 `React.memo`、稳定的 callback、Zustand 的切片订阅，确保拖一个事件不会让一周里其他事件都重渲染。
- **存储**——Dexie 上面包了一层薄薄的 repository。Store 调 repository，组件永远见不到 Dexie。Schema 迁移和 schema 写在一起。

## 关于和 Claude Code 的协作

这个项目是和 [Claude Code](https://www.anthropic.com/claude-code) 一起做的。分工大致是：

- **我负责**：产品方向、架构决策、UX 的主观判断，以及那些和品味相关的部分——配色、字体、决定**不做**什么。
- **Claude 负责**：大部分实现、测试脚手架、以及 pointer events 各种边界情况的耐心调试。

拖拽系统是一个典型例子。第一版用 HTML5 DnD 做原型，跨列拖拽撞到了浏览器原生 API 的墙；第二版改写成 Pointer Events；第三版又过了一遍，把实时预览加上的同时把帧率压在 60fps。每一版都不是"扔一个 prompt 让 AI 写"，而是来回讨论、调整、试错。

仓库里保留了和代码一起长出来的规划文档（`PLAN.md`、`CLAUDE.md`）。
