# CaILens

> 一个克制的周历，正在长成一本时间日志。

CaILens 是一个本地优先的日历应用，灵感来自《奇特的一生》里柳比歇夫坚持了 56 年的"时间统计法"。它的最终目标是成为一个完整的时间日志工具，但**目前的版本只完成了日历部分**——一个干净、好用、本地存储的周视图，接下来才是统计、回顾、分类汇总这些更有趣的部分。

**当前阶段：** 日历模块已经做完。交互参考了 Google Calendar 的周视图（点击创建、拖拽移动、边缘拖拽改变时长、双击编辑），但去掉了你不需要的部分——没有账号，没有云同步，没有提醒，没有任何打扰。

> **状态：** 持续开发中。日历部分已完成，统计视图、日记、导入导出等功能正在路上。

## 为什么做这个

我读《奇特的一生》的时候被柳比歇夫这个人击中了。他从 26 岁开始记录自己每一天每一小时的去向，整整 56 年，直到去世。这件事本身已经够震撼，但更让我在意的是——他不是为了"提高效率"，而是为了**真的看清自己一辈子的时间用在哪里了**。

市面上几乎所有日历工具都在解决另一个问题：把未来安排进格子里。提醒你开会、收会议邀请、设闹钟。但这和"理解自己的时间"是两件事。

CaILens 想做后者。但这条路要走很远，所以第一步先把最基础的工具做扎实——日历的网格本身。这个东西如果用起来不顺手，后面所有的统计、回顾、分类都没意义。

所以这个 README 写在一个尴尬的时间点：产品的"骨架"刚做完，"灵魂"还没开始动工。但它已经能用了，而且我用得挺顺手。

## 现在能做什么

日历模块的交互大致和 Google Calendar 的周视图一致，但有几处取舍不同：

- **24 小时一屏可见**——一周七天每一天都显示完整的 24 小时，不用上下滚动找早上和晚上。
- **点空白格创建**事件。
- **拖动事件移动**，可以跨天拖（这部分写得很费劲，后面会讲）。
- **拖事件的上下边缘改变时长**，不用开任何弹窗。
- **拖动时实时预览**——事件跟着鼠标走，目标时间实时显示，60fps。
- **双击事件改名**，inline 编辑，不开弹窗。
- **右键菜单**——删除、复制、改颜色。
- **六色分类**，让你扫一眼就能看出"这周大致花在哪里了"。
- **本地持久化**——刷新页面、关掉浏览器，数据都还在。

数据存在浏览器的 IndexedDB 里，不上传任何服务器。如果你清空浏览器数据，数据也会一起没——这是本地优先的代价，未来会加导出功能。

## 接下来要做什么

按大致优先级：

- **统计视图**——这才是这个项目真正的核心。柳比歇夫式的分类汇总，可以选任意时间段查看每个分类用了多少小时。日历只是数据的录入界面，统计才是它的归宿。
- **日视图、月视图**——目前只有周视图。
- **重复事件**和模板（比如"早上的固定流程"一键插入）。
- **导入 / 导出**——JSON 和 ICS 格式，让你的数据可以随时带走。
- **全局搜索**所有事件。
- **可选同步**——依然本地优先，同步是 opt-in，不是默认。

如果你对其中任何一项有想法，欢迎开 issue。

## 怎么跑起来

需要 **Node 20 以上**和 **npm**。

```bash
git clone https://github.com/vanemacus486-bit/CaILens.git
cd CaILens
npm install
npm run dev
```

然后打开 Vite 启动后打印的地址（通常是 `http://localhost:5173`）。

### 其他命令

```bash
npm run build       # 类型检查 + 生产构建
npm run preview     # 本地预览构建结果
npm run test        # 跑一次单元测试
npm run test:watch  # 监听模式跑测试
npm run lint        # 跑 ESLint
```

## 架构

代码按层划分，依赖方向严格单向：

- **`domain/`** —— 纯类型和业务规则。不依赖 React、不依赖 IndexedDB、没有副作用。容易单元测试。
- **`data/`** —— Dexie 的 schema 和 repository。整个项目里唯一直接和 IndexedDB 打交道的地方。
- **`store/`** —— Zustand store，包装数据层，对 UI 暴露 selectors 和 actions。
- **`components/`** —— React 组件。永远不直接碰数据层。

几个写得比较费劲的地方：

- **拖拽系统**直接基于 Pointer Events 写，没用现成库。因为我想要的交互组合（跨列拖拽 + 实时预览 + 边缘 resize + 和"点击空白创建"共存）用 HTML5 DnD 或者上层抽象都很难表达干净。命中检测和吸附是对着布局网格算的，不是对着 DOM。
- **渲染性能**——大量用 `React.memo`、稳定的 callback、Zustand 的切片订阅，确保拖一个事件不会让一周里另外 49 个事件都重渲染。
- **存储**——Dexie 上面包了一层薄薄的 repository。Store 调 repository，组件永远见不到 Dexie。Schema 迁移和 schema 写在一起。

## 关于和 Claude Code 的协作

这个项目是和 [Claude Code](https://www.anthropic.com/claude-code) 一起做的。分工大致是：

- **我负责**：产品方向、架构决策、UX 的主观判断（一个交互"感觉对不对"，而不仅仅是"能不能用"），以及那些和品味相关的部分——配色、字体、决定**不做**什么。
- **Claude 负责**：大部分实现、测试脚手架、以及 pointer events 各种边界情况的耐心调试。

拖拽系统是一个典型例子。第一版用 HTML5 DnD 做原型，跨列拖拽撞到了浏览器原生 API 的墙；第二版改写成 Pointer Events；第三版又过了一遍，把实时预览加上的同时把帧率压在 60fps。每一版都不是"扔一个 prompt 让 AI 写"，而是来回讨论、调整、试错。

如果你对这个工作流好奇，仓库里保留了和代码一起长出来的规划文档（`PLAN.md`、`CLAUDE.md`）。
